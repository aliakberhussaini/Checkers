# Neon Checkers v1.0 — Session Build Log

Human vs. an IBM MAPE-K autonomic agent, in the browser, zero dependencies.
This is a record of *how* it was built and *why* each decision was made —
not a user-facing README (see [README.md](README.md) for that).

## 1. The ask

Build a complete checkers game: refreshing graphics, one human player, one
AI/agent player, and the AI must be a classic **MAPE-K** control loop (IBM
autonomic computing, Kephart & Chess 2003) that learns the specific human
opponent across games and uses that to beat them — never a static bot.

## 2. Approach: contract first, then parallel workflow

Before writing any game code, I wrote `SPEC.md` — rules, state/move shapes,
exact `CheckersEngine`/`CheckersAI` APIs, the MAPE-K learning requirements,
and a UI contract. This became the binding contract every subsequent agent
built against, so parallel work couldn't drift into incompatible shapes.

Build ran as a `Workflow` (multi-agent orchestration), phased:

1. **Engine** — built and tested alone first, since the AI's tests need to
   `require()` it.
2. **Build** — MAPE-K agent and frontend built **in parallel**, disjoint file
   ownership (`ai.js`+tests vs. `index.html`/`styles.css`/`ui.js`), both
   coding strictly against `SPEC.md` rather than against each other.
3. **Integrate** — one agent ran every test, wrote a 5-game headless
   integration test (scripted greedy "human" vs. the persistent agent), and
   the README.
4. **Review → adversarially verify → fix** — four review lenses (rules,
   MAPE-K fidelity, frontend, cross-file integration) each ran read-only;
   every finding then went to a *separate* verifier agent instructed to try
   to **refute** it with a repro or mutation test before it counted as real.
   Only survivors reached the fixer.

That review→verify→fix pattern is the single most valuable structural
decision in this session — plain code review would have accepted several
findings that turned out to be non-issues, and would have missed others
that only a mutation test could expose (see §4).

## 3. First delivered version: "refreshing" aqua theme

Engine: 8×8 American checkers, mandatory captures, fully-expanded multi-jump
sequences, one-step kings, draw at 80 quiet half-moves. Pure `applyMove`,
stable `hash()`.

AI (`ai.js`): five explicit classes — `Monitor`, `Analyzer`, `Planner`,
`Executor`, `KnowledgeBase` — that **only** communicate by reading/writing
the shared `KnowledgeBase`, never by calling each other directly. That
discipline is what makes it a real MAPE-K loop instead of a minimax bot
with cosmetic labels. Learning is real, not decorative: a ≥4-ply blunder
check, a player profile (`exposureRate`, `edgePreference`, `advanceRate`,
`kingUsage`, `blunderRate`) with sample-size confidence, and an opening
book keyed by board hash. Each profile trait crossing a threshold changes
a concrete Executor parameter — trap-seeking weight, center weight, search
depth, or opening-move bias — never just a display label.

Frontend: aqua/mint gradient, frosted-glass panels, coral vs. charcoal
pieces, five-section live MAPE-K panel, animated multi-jump paths.

Result: 60+57+12+17 tests green; the agent beat a scripted greedy opponent
5-0 in the integration test. One review round ran mid-build and hit the
session's usage limit before finishing — resumed later from the workflow's
cached journal (`resumeFromRunId`), which replayed every already-completed
agent call instantly and only re-ran the interrupted review/verify/fix
phases live. That resumability is what made the mid-session interruption a
non-event instead of a redo.

## 4. Bugs the adversarial-verify pass actually caught

Real, verifier-confirmed defects from that first round (each fixed):

- **Cyclic multi-jump risk**: mutation-testing showed a plausible refactor
  (searching the board array in place instead of lifting the moving piece
  off its origin square) would pass all existing tests yet corrupt state on
  a specific king cycle — added a test that only a correct implementation
  passes.
- **`legalMoves` purity was asserted only indirectly.** A mutant that
  searched in place (no defensive copy) passed the full suite but mutated
  the caller's board. Added a direct before/after deep-equality test.
- **Forced-capture enumeration** was only ever tested with one capturing
  piece on the board at a time — a mutant returning only the first jumper's
  moves passed everything. Added a two-simultaneous-jumper case.
- **A king reaching the back row wrongly could have reset the quiet
  counter** if the `!piece.k` promotion guard were ever weakened — added a
  regression test pinning that a king's own back-row arrival does not
  reset it.

These are the kind of bugs that "the tests pass" hides and "does it look
right" never surfaces — they only show up when someone tries to break the
invariant on purpose.

## 5. The design pivot: user-supplied Claude Design bundle

Partway through, the user supplied a Claude Design project ("Neon Checkers",
grid variant) via `claude_design`/DesignSync. That tool needs an interactive
`/design-login`, unavailable in this non-interactive session — the user
instead exported the bundle manually to
`checkers-game-design-web-and-mobile/` inside the project.

Decision: treat the prototype's HTML/CSS purely as a **visual and behavioral
spec**, and discard its internal game logic entirely. `engine.js` and
`ai.js` — already built, tested, and adversarially reviewed — stayed
untouched; only `index.html`, `styles.css`, and `ui.js` were rewritten from
scratch to reproduce the design (dark neon theme, glowing cyan/violet
pieces, turn timer, hint/undo, WebAudio sfx, settings modal, title/game-over
overlays) while wiring every interaction to the real engine and the real
agent instead of the prototype's placeholder one-ply scorer.

Two race conditions were fixed proactively during that rewrite, before any
review flagged them, because they were foreseeable from the design's own
timing (`schedule()`-based phase animations):
- the blunder-search inside the `playerMove` monitor event can take up to
  ~250ms; deferred it one tick so it can never stall the move's paint.
- if "New Game" fires while the winning move is still animating, the
  decided game's `gameEnd` event must still reach the agent's knowledge
  before state resets — added `flushPendingGameEnd()`.

## 6. Second review round: backlog + design fidelity

A follow-up workflow first applied 9 pre-confirmed findings against `ai.js`
(dead-recorded book fields now actually consulted; a bias-sum invariant
that could exceed a man's value, capped; forced moves no longer inflating
the blunder-rate metric; `trapSeeking`/`useOpeningBook` flags rewritten to
be *derived from* the mechanisms that fire rather than set independently —
so the insights panel can never silently diverge from real behavior; a
stale-plan cache cleared on `gameStart`), then ran a **design-fidelity**
lens (line-by-line against the `.dc.html` source) alongside a repeat
functional lens on the new UI.

Confirmed and fixed: MAPE phase names weren't reflected in the status
sub-text; capture counts visually lagged the board during multi-jump
chains; Google Fonts links violated the project's own zero-external-URL/
offline requirement (removed, falls back to system fonts); "Easy" mode was
silently teaching the agent from moves it played randomly, polluting the
opening book (now gated off); a MAPE phase chip could stay lit after an
interrupted game; and the SPEC's five-section panel / mandatory-capture
notice / draw count — dropped in the visual redesign — were restored as a
new `mape-live` panel that reads `agent.getInsights()` directly.

## 7. Post-delivery bug the user caught by actually reading the panel

After delivery, the user pointed at the live MAPE-K panel and said the
content wasn't showing. Root cause: `.mape-val` in `styles.css` had
`white-space: nowrap; overflow: hidden; text-overflow: ellipsis` — every
reasoning sentence ("Player exposes pieces on 49% of moves -> trap-seeking
bonus enabled...") was silently clipped to one line with "…". Every
automated check (syntax, DOM-id presence, field-mapping audits) had passed,
because none of them read the rendered text for legibility. Fixed to wrap
(`white-space: normal; overflow-wrap: break-word`) and confirmed via an
accessibility-tree snapshot that the full sentence renders.

**Lesson**: structural review (does the element exist, is the field wired)
is not the same check as "can a human actually read this." The only thing
that caught it was the user looking at the screenshot.
